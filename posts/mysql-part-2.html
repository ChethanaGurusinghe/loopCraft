What is an entity ? (Strong Entity)

An entity can be a person, a place, or an event. An entity is represented by a rectangle. The name of this entity must be singular. That is, there cannot be an entity called Students.

Also, a table is created in the database for each Entity in ER. There are types of entities. Normally, an entity is drawn like this. We call it a Strong Entity. There are also Strong and Weak entities.

Week Entity

A week entity is represented as a rectangle within a rectangle. Employee children depend on the employee. Such entities are called week entities.

The second part of an ER is called attributes.‚Ä¶.

What is an attribute ?

The things used to describe an Entity in an ER are called attributes. As an example, if we consider the entity Student, how can we describe the Student?
The student has things like a name, age, date of birth, and address. These things are called attributes. We can represent it like this.

From a database perspective, the columns in the Student table are the attributes in the ER.

There are 4 types of attributes.

  1. Simple attributes
  2. Composite attributes
  3. Derived attributes
  4. Multivalued attributes

(01.) Simple Attributes

That is, like we talked about before, a normal attribute.

(02.) Composite Attributes

A composite attribute is an attribute with other attributes...it may not be clear to say that. Let's see with an example.

You can read it now until you get to the point where it says that the student has a name. A name can have two parts: first name and last name (surname). That is what is meant by having other properties. If you don't understand, let's look at another example.

Simply put, an address is made up of three parts: country, city and street. (It can vary from ER to ER, depending on the purpose )

(03.) Derived Attributes

We can get the value of this attribute type even if we use another attribute. For example, if we have the date of birth (dob) stored in the database, we can calculate the age using that. Attributes like this are called derived attributes.

(04.) Multivalued Attributes

Multivalued Attributes are those that have multiple values. An example is a phone number. Since a person can have more than one phone number, it can be said to be Multivalued.

The 3rd part of an ER is the relationship‚Ä¶.

What is a relationship in an ER ?

A relationship is a connection between two entities. That is, between two tables in a database. When drawing an ER Diagram, a relationship is represented as follows.

This symbol contains a verb. That verb should match the entities. I will explain how a relationship and an entity are related.
If there is a relationship between two entities, it must have a cardinality. Depending on that cardinality, there are 3 types of relationships.

  1. One to One (1:1)
  2. One to Many (1:M)
  3. Many to Many (M :N)

(01.) One to One Relationship (1:1)

This is a 1:1 relationship. Otherwise, it is also called a relationship.

Why is it called a 1:1 relationship?

In the example image I posted, it is now clear to you that there are 2 entities and a relationship between them. If you read the relationship, each country has a country code. To clearly understand the cardinality, you have to consider both entities.

A country can only have one country code, and only one country can use one country code.

In this way, the cardinality can be easily determined by reading the relationship between two entities in the ER Diagram.

(02.) One to Many Relationship (1:M)

Both 1:M and M:1 are considered the same.

We can read this as a 1:1 relationship.

There can be many students in one class. A student can only be in one class.

(03.) Many to Many Relationship (M:N)

One student can study many subjects. There can be many students studying one subject.

Why is it M:N instead of M:M in a Many to Many relationship?

If we put both sides of a Many to Many as M:M, it means that there are many on both sides but they are the same. That means if there are 10 students, there are also 10 subjects. That is not what we wanted to mean. There are many on both sides. Those quantities can be unequal. That is why it is M:N. You will understand this very well if you think of it in terms of 1:1. Since there are equal quantities on both sides (1), it is 1:1. We talked about cardinality. Cardinality can be expressed in the same way. 1:1, 1:M, M:N are types of Chen‚Äôs notation. Another type is Crow foot notation. In this, we use two types of mandatory and optional. If mandatory, the symbol | is used, and if optional, the symbol 0 is used. Here one is represented by 1 as normal. Many is represented by < this symbol.

This is how to distinguish between mandatory and optional.

Does a class need to have a teacher?

Since there can be teachers without a class, it is not necessary for a teacher to have a class. Therefore, it is optional (0). Since a class must have a teacher, it is mandatory (|).
